## Prioritized Backlog

### 1. High: local workspace runtime can fail at `/` with a blank internal-error screen

Problem statement:
The current workspace runtime is unstable enough that the app can fail before any UX flow starts. This blocks independent validation and creates a severe confidence gap for anyone trying to verify the current branch locally.

Severity:
High

Reproduction:
1. Start the current workspace locally.
2. Open `http://127.0.0.1:3012/` with `agent-browser`.
3. Observe a blank page with only `Internal Server Error`.
4. In the dev server logs, the page fails around missing `.next/routes-manifest.json` and `.next/server/pages/_document.js`.

Evidence:
- `docs/qa/screenshots/ux-audit-2026-03-26/00-desktop-current.png`

### 2. High: mobile navigation keeps the drawer open while the destination view updates behind it

Problem statement:
On mobile, choosing a destination from the drawer can leave the navigation panel covering the page even after the underlying content changes. That makes the transition ambiguous and forces an extra mental step because the user does not get a clean arrival state.

Severity:
High

Reproduction:
1. Open the shipped instance at `390x844`.
2. Sign in.
3. Open the mobile navigation drawer.
4. Tap `Katalog`.
5. Observe that the destination state changes, but the drawer remains on top instead of yielding the page cleanly.

Evidence:
- `docs/qa/screenshots/ux-audit-2026-03-26/10-mobile-nav-open.png`
- `docs/qa/screenshots/ux-audit-2026-03-26/11b-catalog-mobile-correct.png`

### 3. Medium: mobile login submit action sits below the fold after the user enters credentials

Problem statement:
The mobile auth screen spends a large amount of vertical space on hero copy, locale controls, and tab controls before the primary submit action. A returning user can fill email and password without seeing the actual login CTA, which increases friction for a simple repeat task.

Severity:
Medium

Reproduction:
1. Open the auth screen on a `390x844` viewport.
2. Switch to `Anmelden`.
3. Fill email and password.
4. Observe that the submit button is below the visible portion of the form until the user scrolls.

Evidence:
- `docs/qa/screenshots/ux-audit-2026-03-26/07b-mobile-login-below-fold.png`

### 4. Medium: non-dashboard pages inherit the dashboard hero and KPI strip, weakening page identity and pushing primary tasks down

Problem statement:
Catalog, profile, rules, and print flows still carry the `Was jetzt Aufmerksamkeit braucht.` hero and KPI strip above their own feature content. This makes every area feel like a continuation of the dashboard rather than a distinct task surface, and it pushes the real working UI lower on the page.

Severity:
Medium

Reproduction:
1. Sign in on desktop.
2. Visit `Katalog`, `Profile`, `Regeln`, or `Druckbögen`.
3. Observe that the dashboard hero and summary KPI row still occupy the top of the page before the actual section-specific UI begins.

Evidence:
- `docs/qa/screenshots/ux-audit-2026-03-26/02b-catalog-desktop-clean.png`
- `docs/qa/screenshots/ux-audit-2026-03-26/03b-profiles-desktop-clean.png`
- `docs/qa/screenshots/ux-audit-2026-03-26/04-rules-desktop.png`
- `docs/qa/screenshots/ux-audit-2026-03-26/06-sheets-desktop.png`

### 5. Medium: German profile form uses US-style date placeholders

Problem statement:
The German-first profile setup screen shows `mm/dd/yyyy` inside the date fields. That conflicts with the rest of the German UI and creates avoidable uncertainty about the expected date format.

Severity:
Medium

Reproduction:
1. Open `Profile` on desktop.
2. Inspect `Datum letzter Frost` and `Datum erster Frost`.
3. Observe the `mm/dd/yyyy` placeholder pattern in a German interface.

Evidence:
- `docs/qa/screenshots/ux-audit-2026-03-26/03b-profiles-desktop-clean.png`

Low
